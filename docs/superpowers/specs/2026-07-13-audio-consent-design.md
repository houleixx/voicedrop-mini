# VoiceDrop 小程序音频信息授权设计

日期：2026-07-13

## 目标

在微信小程序所有采集音频的功能之前增加独立、明确、可撤回的《音频信息授权协议》授权门槛。只有用户主动同意当前协议版本后，小程序才能申请微信麦克风权限、开始录音或上传音频。

协议只描述产品真实存在的处理：录音、上传、语音转文字、文章生成和编辑、语音指令及社区语音回应。当前系统不提取声纹模板，不进行声纹比对或声纹身份认证，因此协议不得声称存在这些能力；同时明确提醒音频中可能包含能够识别个人的声音特征。

## 范围

授权门槛覆盖所有当前麦克风入口：

1. 首页轻点开始主录音，以及录音页内可选的 AI 采访。
2. 首页长按发起语音指令。
3. 文章详情页长按发起语音修改。
4. 社区详情页录制语音回应。

本次不修改后端 API、WebSocket 消息、认证、录音文件名、图片 marker、上传格式或 Android/iOS 客户端。Android 和 iOS 不受代码影响；如果其上架审核也要求独立音频授权，需要另行同步相同语义。

## 用户体验

### 首次授权

用户首次触发任一音频功能时，小程序先展示独立授权确认，而不是直接调用 `wx.authorize`：

- 标题为“音频信息授权”。
- 摘要说明将录制并上传用户主动提供的音频，用于转写、生成或编辑文章、执行语音指令以及生成社区回应。
- 提示音频可能包含能够识别个人的声音特征，但 VoiceDrop 不提取声纹模板，也不将音频用于声纹身份识别、广告画像或向无关第三方出售。
- 提供“查看完整协议”“不同意”“同意并继续”三个明确动作。

“查看完整协议”进入独立协议页。用户返回原页面后仍未被视为同意，必须再次触发语音功能并明确点击“同意并继续”。

“不同意”立即终止当前操作；不得申请麦克风权限、创建录音会话、连接语音 WebSocket 或上传任何音频。

“同意并继续”在本机保存当前协议版本和同意时间，然后继续现有的微信麦克风权限流程。微信权限被拒绝时，沿用各入口现有的“去设置”提示；业务协议同意状态与微信系统权限状态相互独立。

### 后续使用与协议升级

已同意当前协议版本的用户再次触发音频功能时，不重复显示业务授权确认，直接进入微信权限检查和原有业务流程。

授权状态按协议版本保存。协议正文发生实质变化时提升版本号；旧版本授权自动失效，用户下次使用音频功能时需要重新同意。

### 查看与撤回

“关于”页新增《音频信息授权协议》长期入口和“撤回音频授权”操作。

撤回只清除本机保存的协议版本和同意时间，不自动删除历史录音。撤回确认文案明确说明：

- 撤回后再次使用任何语音功能时需要重新授权。
- 已有录音和处理结果不会因此自动删除。
- 用户可以在录音列表删除单条数据，或在账户页注销并永久删除全部数据。

## 协议正文要求

独立协议页面至少包含以下内容：

1. **信息类型**：用户主动录制的音频、录音时长和与本次功能直接相关的处理结果；音频中可能包含可识别个人的声音特征。
2. **收集目的**：把口述转成文字和文章、按语音指令编辑内容、支持 AI 采访、生成社区回应。
3. **处理方式**：在用户主动操作后调用设备麦克风，将音频或实时音频帧通过加密网络发送至 VoiceDrop 后端，执行自动化语音处理，并按具体功能保存录音及处理结果。
4. **使用范围**：仅用于用户主动触发的 VoiceDrop 功能；不提取声纹模板，不进行声纹身份识别，不用于广告画像，不向无关第三方出售。
5. **存储与删除**：音频和处理结果随用户当前 VoiceDrop 账户保存；用户可删除单条录音，或通过注销账户删除云端与本机的全部数据。传输失败时不得扩大收集范围。
6. **用户权利**：用户可以拒绝或撤回授权；拒绝或撤回只影响音频相关功能，不影响浏览已有内容等无需麦克风的功能。
7. **联系方式**：沿用项目现有支持与投诉邮箱。
8. **生效日期和版本**：显示 `2026-07-13` 和与代码中一致的协议版本。

协议文案使用“音频信息授权”，避免把普通录音和语音转写错误描述成已实施的生物识别或声纹认证能力。

## 组件与职责

### `utils/audio-consent.js`

统一管理协议元数据和本机授权状态：

- 导出当前协议版本、生效日期、标题及正文。
- 判断本机是否已同意当前版本。
- 写入当前版本和同意时间。
- 撤回本机授权。
- 提供统一的授权确认流程，使调用方只在成功回调或已解决的成功结果后继续申请 `scope.record`。

该模块不申请麦克风权限，也不启动具体录音，避免把业务授权与微信平台权限混在一起。

### `pages/audio-consent/index`

只负责完整、可滚动地展示协议正文和版本信息。页面本身不写入同意状态；仅仅阅读协议不等于授权。

### 各音频入口

每个入口先调用统一授权模块，通过后再执行原有 `wx.authorize({ scope: 'scope.record' })` 和录音逻辑。授权结果晚于手势结束返回时，现有长按录音入口仍必须保持防止“松手后才开始录音”的保护。

### `pages/about/index`

增加协议入口、当前授权状态和撤回操作。现有隐私说明继续保留，用于说明更广泛的数据同步；音频授权协议作为独立条目存在。

## 数据流

```text
用户触发语音功能
  → 检查当前协议版本是否已同意
    → 未同意：显示授权确认
      → 查看协议：打开独立页面，当前操作终止
      → 不同意：当前操作终止
      → 同意：保存版本与同意时间
    → 已同意：继续
  → 申请/检查微信 scope.record 权限
    → 拒绝：提示去设置，当前操作终止
    → 允许：进入原有录音、实时处理或上传流程
```

本机仅新增保存协议版本和同意时间。音频的后端存储位置、访问令牌隔离、上传接口和删除接口均保持现状。

## 异常与边界行为

- 本机存储读取失败时按“未授权”处理，不得开始录音。
- 本机存储写入失败时不得视为已授权；提示用户授权状态保存失败并终止本次录音。
- 用户在授权弹窗、协议页或微信权限弹窗中取消时，不得创建录音所有权状态或遗留 WebSocket 会话。
- 多次快速触发时，同一页面不得并发显示多个授权流程或启动多个录音会话。
- 撤回发生在录音进行中时不强制截断已开始的录音；撤回入口位于“关于”页，正常导航到该页前现有录音流程已按页面生命周期处理。撤回影响下一次音频采集。
- 协议升级只使旧授权失效，不删除历史数据。

## 测试与验收

Node 自动化测试覆盖：

1. 未同意、已同意、撤回和协议版本升级后的状态判断。
2. 用户拒绝时不调用麦克风权限和业务继续函数。
3. 用户查看协议时正确导航，且不写入同意状态。
4. 用户同意并成功写入状态后才继续；写入失败时终止。
5. 主录音、首页语音指令、文章语音修改和社区语音回应四个入口均先经过统一授权。
6. “关于”页能打开协议并撤回授权。
7. 独立页面已在 `app.json` 注册，结构和资源引用通过小程序校验。

交付前运行：

```bash
npm test
npm run validate:miniapp
```

微信开发者工具和真机还需人工验证：

- 首次同意、拒绝、查看协议后返回、撤回后再次触发。
- 微信麦克风权限首次申请、拒绝、从设置恢复。
- 四个音频入口均不会在业务授权之前采集音频。
- 协议页在常见屏幕尺寸上可完整滚动阅读。

## 完成标准

- 小程序内存在可长期访问的独立《音频信息授权协议》。
- 所有音频采集入口在用户明确同意当前版本前均无法启动。
- 协议准确说明音频的目的、方式和用途，不虚构声纹识别能力。
- 用户可以撤回授权，且撤回与历史数据删除的差异被明确说明。
- 自动化测试与小程序结构校验通过，并记录所需的真机验证项。
