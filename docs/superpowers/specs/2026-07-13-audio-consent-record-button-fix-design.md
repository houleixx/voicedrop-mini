# 录音按钮音频授权修复设计

## 背景

当前录音入口通过页面的 `selectComponent()` 获取音频授权弹窗。组件未被找到或未暴露 `request()` 时，页面直接返回拒绝结果，随后静默结束录音流程。用户首次点击录音按钮时因此可能既看不到授权协议，也看不到错误提示。

## 目标行为

1. 用户首次点击需要音频的功能时，显示独立的音频信息授权弹窗。
2. 用户点击“同意并继续”后，保存当前协议版本及同意时间到本地，并立即继续原操作。
3. 本地存在当前版本的有效同意记录时，后续操作不再显示弹窗。
4. 用户点击“不同意”时不保存授权、不继续原操作；下次点击时再次显示授权弹窗。
5. 授权记录缺失、读取失败、被撤销或协议版本升级时，均视为未授权。
6. 授权流程异常时显示明确提示，不允许静默无响应。

## 方案

采用页面状态驱动的受控弹窗，不再由页面命令式调用子组件方法。

- 页面调用授权请求函数时，先通过 `utils/audio-consent.js` 检查本地版本化状态。
- 已授权时立即返回成功结果。
- 未授权时，页面设置弹窗可见状态，并保存当前待完成请求的 Promise resolver。
- 弹窗通过 `agree`、`decline` 事件把用户选择返回页面。
- 页面收到同意事件后保存授权并完成 Promise；收到拒绝事件后仅完成 Promise，不写本地状态。
- 页面继续沿用现有的微信麦克风权限申请和录音/语音功能逻辑。
- 重复点击共享同一个待处理请求，避免出现多个弹窗或覆盖 resolver。
- 页面卸载时以拒绝结果完成未结束请求，避免悬挂 Promise。

弹窗组件只负责展示和发出用户动作事件；本地授权状态与待执行操作由页面负责协调。录音首页、详情页语音编辑和社区语音回应使用相同接口语义。

## 错误处理

- 本地读取异常按未授权处理并显示弹窗。
- 本地写入失败时保持未授权，关闭当前请求并显示“授权状态保存失败”。
- 弹窗事件或页面状态异常时显示“授权组件加载失败，请重试”，不进入微信麦克风权限申请。
- 用户拒绝属于正常选择，不显示错误提示。

## 测试

自动化测试覆盖：

- 首次请求显示弹窗，且尚未调用微信麦克风权限。
- 同意后写入当前协议版本并继续原操作。
- 已同意时跳过弹窗。
- 拒绝时不写本地状态，下次请求再次显示弹窗。
- 重复请求复用同一个待处理 Promise。
- 页面卸载会结束待处理请求。
- 本地写入或弹窗协调异常时显示提示并停止后续操作。

交付前运行 `npm test` 和 `npm run validate:miniapp`。此外需在微信开发者工具与真机验证首次同意、拒绝后重试、再次启动小程序后免提示，以及撤销授权后重新提示。

## 范围与跨端影响

本次仅修改微信小程序的授权弹窗协调方式，不改变协议正文、后端接口、音频上传、录音文件名或跨端数据契约。Android、iOS 和后端无需同步修改。
